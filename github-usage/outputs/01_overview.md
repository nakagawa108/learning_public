# GitHubの使い方 概要

## 1. この教材で学ぶこと
この教材では、GitHubを使って「変更を残す」「変更を共有する」「変更を提案する」という3つの基本を学びます。

最初に大事な区別をします。

- Git: 変更履歴を記録する仕組み
- GitHub: Gitで管理した内容をオンラインで共有しやすくするサービス

例:
レポートを何度も書き直すとします。Gitは「どこを直したか」を残す記録帳です。GitHubは、その記録帳を置いて人と見せ合う共有場所です。

要点:
Gitは記録の仕組み、GitHubは共有の場です。

## 2. まず覚える流れ
GitHubの基本利用は、次の流れで考えると分かりやすいです。

1. GitHubでリポジトリを用意する
2. `clone` して自分のPCに持ってくる
3. ファイルを編集する
4. `commit` で変更を記録する
5. `push` でGitHubへ送る
6. 必要なら Pull Request で提案する

短い例:

```bash
git clone https://github.com/example/study-notes.git
cd study-notes
git commit -m "READMEを更新"
git push origin main
```

この例は「持ってくる」「記録する」「送る」を並べたものです。今は細かい文法より、流れだけつかめれば十分です。

要点:
GitHub学習では、操作名より先に流れを覚えると理解しやすいです。

## 3. 先に覚えたい用語
ここでは最初によく出る言葉だけを短く確認します。

- リポジトリ: ファイルと変更履歴をまとめて管理する場所
- clone: GitHub上のリポジトリを自分のPCにコピーすること
- commit: 変更をひとかたまりとして記録すること
- push: ローカルの記録をGitHubへ送ること
- Pull Request: 変更を取り込んでほしいと提案すること

例:
`README.md` を直したあとに `commit` すると、その修正が履歴として残ります。さらに `push` すると、その履歴がGitHubでも見られるようになります。

要点:
用語は単体で覚えるより、流れの中で覚えるほうが定着しやすいです。

## 4. サンプルコードをどう読むか
この教材では短いコマンド例をたくさん使います。コードを全部暗記する必要はありません。

見るポイント:
- その行は何をしたいのか
- どこがローカル作業で、どこがGitHubとのやり取りか
- 前の行と次の行で役割がどう変わるか

例:

```bash
git add README.md
git commit -m "READMEを更新"
```

1行目は「変更を記録対象として選ぶ」操作です。2行目は「その変更を履歴として残す」操作です。

要点:
コードは丸暗記より、「この行の役割は何か」で読むのが大切です。
