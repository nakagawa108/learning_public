# GitHubの使い方 レッスン

## 1. GitとGitHubの違い
定義:
Gitは変更履歴を管理する仕組み、GitHubはその内容をオンラインで共有しやすくするサービスです。

例:
作文を何度も書き直すとき、Gitは「何を直したか」を残す記録帳です。GitHubは、その記録帳をクラスのみんなで見られる棚のようなものです。

混乱しやすい例:
- 悪い理解: GitHubが全部の履歴を自動で管理してくれる
- 正しい理解: 履歴を作る中心はGitで、GitHubはそれを見たり共有したりしやすくする

要点:
Gitは記録、GitHubは共有です。

## 2. リポジトリとは何か
定義:
リポジトリは、ファイルと変更履歴をまとめて管理する場所です。

例:
自己紹介サイトを作るなら、HTML、CSS、画像、更新履歴を1つの箱にまとめます。その箱がリポジトリです。

短いイメージ例:

```text
my-profile/
  README.md
  index.html
  style.css
```

これは「どんなファイルを管理する箱か」を表した例です。まだGitHubへ送っていなくても、リポジトリという考え方は同じです。

要点:
リポジトリは「プロジェクトの箱」です。

## 3. ローカル作業とGitHub上の作業
定義:
ローカルは自分のPC上の作業、GitHub上はオンラインの共有場所での状態を指します。

例:
`README.md` をPCで書き換えるのはローカル作業です。`push` してはじめてGitHub上の内容も更新されます。

短いコード例:

```bash
notepad README.md
git status
```

1行目はファイル編集の例です。2行目は、編集した結果をGitがどう見ているかを確認する例です。

混乱しやすい例:
- 悪い理解: PCで保存したからGitHubも更新された
- 正しい理解: PCで保存しただけでは、まだGitHubは変わらない

要点:
ローカルとGitHub上の状態は、自動では同じになりません。

## 4. `clone` の役割
定義:
`clone` は、GitHub上のリポジトリを自分のPCにコピーする操作です。

まず短い例を見ます。

```bash
git clone https://github.com/example/study-notes.git
cd study-notes
```

1行目はGitHub上のリポジトリを手元へ持ってくる操作です。2行目は、その作業フォルダへ移動する操作です。

もう1つの見方:

```bash
git clone https://github.com/example/study-notes.git my-notes
```

この例では、手元のフォルダ名を `my-notes` にして作成しています。最初は意味が分かれば十分で、書き方を全部覚える必要はありません。

要点:
`clone` は作業開始の入口です。

## 5. 変更を確認する `status`
定義:
`git status` は、今の変更状態を確認するためのコマンドです。

短い例:

```bash
git status
```

これは「何が変わったか」「まだ記録していない変更があるか」を確認するために使います。

出力の見方のイメージ:

```text
modified: README.md
```

この表示は、`README.md` が変更されたことを表します。初心者はまず「どのファイルが変わったか」を読めれば十分です。

要点:
作業中に迷ったら、まず `git status` で今の状態を確認します。

## 6. `commit` の役割
定義:
`commit` は、変更をひとかたまりとして記録する操作です。

まず短い例です。

```bash
git add README.md
git commit -m "READMEの自己紹介を更新"
```

1行目は、記録したい変更を選ぶ操作です。2行目は、その変更を説明つきで履歴に残す操作です。

もう少し具体的な例:

```bash
git add index.html style.css
git commit -m "見出しと色を調整"
```

この例では、複数ファイルの変更を1つのまとまりとして記録しています。同じ目的の変更を1つにまとめると、あとで履歴を読みやすくなります。

混乱しやすい例:
- 悪い理解: `commit` はただの上書き保存
- 正しい理解: `commit` は意味のある変更を履歴として残す操作

要点:
`commit` は変更を説明つきで残す記録です。

## 7. `push` の役割
定義:
`push` は、ローカルで記録した `commit` をGitHubへ送る操作です。

短い例:

```bash
git push origin main
```

この行は、今ある記録を `origin` という接続先の `main` ブランチへ送る、という意味です。最初は「GitHubへ送る操作」と読めれば大丈夫です。

流れで見る例:

```bash
git add README.md
git commit -m "READMEを更新"
git push origin main
```

1行目で変更を選び、2行目で記録し、3行目でGitHubへ送っています。この3段階を分けて理解するのが大切です。

要点:
`commit` は記録、`push` は共有です。

## 8. Pull Request の役割
定義:
Pull Request は、「この変更を取り込んでください」と提案する仕組みです。

例:
チームの手順書を直したあと、すぐ本体に反映するのではなく、まず変更内容を見てもらいたいとします。そのときに Pull Request を使います。

見るポイント:
- どのファイルを変えたか
- なぜ変えたか
- コメントや確認結果がどうなっているか

イメージ例:

```text
タイトル: READMEの説明をわかりやすく修正
内容: 初学者向けに用語説明を追加
```

これは Pull Request で伝える情報の短い例です。コードだけでなく、意図も共有できるのが大切です。

要点:
Pull Request は変更提案と確認の場です。

## 9. 最小の流れを通して見る
最後に、最小の流れを1本で見ます。

```bash
git clone https://github.com/example/study-notes.git
cd study-notes
git status
git add README.md
git commit -m "READMEを更新"
git push origin main
```

読み方:
- 1行目: 手元へ持ってくる
- 2行目: 作業場所へ入る
- 3行目: 今の状態を見る
- 4行目: 記録したい変更を選ぶ
- 5行目: 履歴として残す
- 6行目: GitHubへ送る

この6行を1回の物語として読めるようになると、GitHubの基本利用がかなり分かりやすくなります。

要点:
基本は「持ってくる → 編集する → 記録する → 送る」です。

## 10. 最初の学習で意識したいこと
最初から全部を覚える必要はありません。まずは次の3点を言えるようになれば十分です。

- Gitは履歴管理、GitHubは共有サービス
- 変更は `commit` で記録し、`push` で送る
- Pull Request は変更を取り込んでもらうための提案

要点:
最初の目標は、用語の区別と基本の流れをつかむことです。
