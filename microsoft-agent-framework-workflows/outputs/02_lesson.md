# Microsoft Agent Framework Workflows レッスン

## 1. Workflow とは何か
Workflow とは、複数の処理ステップをつないで、流れとして実行する仕組みです。

まずは一番小さいイメージを見ます。

```text
入力
  ↓
要約する
  ↓
レビューする
  ↓
結果を返す
```

この流れでは、1 回のモデル呼び出しではなく、役割の違う複数ステップが並んでいます。  
こうした「順番のある処理」を見通しよく設計するために Workflow を使います。

Agent だけでも複数回 `run()` して似たことはできます。  
ただし、分岐、並列、途中停止、再開まで含めると、Workflow として構造化したほうが読みやすく、保守もしやすくなります。

### 要点
- Workflow は処理の流れを設計する仕組みです。
- 複数ステップがあるときほど効果が出ます。

## 2. Agent と Workflow の違い
Agent は 1 つの判断や応答を担当し、Workflow はそれらをどう並べるかを担当します。

たとえば、次のように考えると区別しやすいです。

- Agent: 「文章を要約する担当」
- Workflow: 「要約したあと、レビューして、条件次第で人に確認を取る流れ」

悪い理解の例:
- Workflow は Agent を何回か呼ぶだけのもの

よい理解の例:
- Workflow は、どの順で動かすか、どこで分岐するか、どこで止めるかを表す設計

Microsoft Agent Framework では、Agent と Workflow を別の軸として学べるよう整理されています。  
そのため、「賢い 1 ステップ」と「複数ステップの制御」は分けて考えると理解しやすくなります。

### 要点
- Agent は実行担当です。
- Workflow は制御担当です。

## 3. Workflow の基本部品
Workflow を読むときは、まず `Executor`、`Edge`、`WorkflowBuilder` を見ます。

### Executor
Executor は、Workflow の中の 1 ステップを担当する実行単位です。

例:
- テキストを要約する
- 分類する
- 承認が必要か判定する
- 複数結果をまとめる

### Edge
Edge は、処理が次にどこへ流れるかを表す接続です。

公式ドキュメントでは、代表的な Edge パターンとして次が案内されています。

- Direct
- Conditional
- Switch-case
- Fan-out
- Fan-in

### WorkflowBuilder
WorkflowBuilder は、Executor と Edge を組み立てて 1 つの Workflow を作るための部品です。

### 要点
- Executor は「何をするか」です。
- Edge は「どこへ進むか」です。
- WorkflowBuilder は「どう組み立てるか」です。

## 4. 最小の直列 Workflow を読む
最初は、直列実行だけの Workflow から入るのがわかりやすいです。

次のコードは、入力を「要約」してから「レビュー」する流れを表した概念例です。

```python
from agent_framework import WorkflowBuilder

builder = WorkflowBuilder()

builder.add_executor("summarize")
builder.add_executor("review")
builder.add_edge("summarize", "review")

workflow = builder.build()
result = workflow.run("この文章を短く整理してからレビューしてください。")
print(result)
```

このコードで見たいポイントは 3 つです。

1. `summarize` と `review` が別ステップであること
2. `add_edge()` で流れを明示していること
3. 最後に `build()` して実行すること

実際のプロジェクトでは、文字列名ではなく、Executor オブジェクトやハンドラー関数を登録する形になることがあります。  
ただし初学者の段階では、「まずノードを置き、次に線をつなぐ」と捉えると十分です。

### 要点
- 直列 Workflow は一番小さい基本形です。
- ノードを置いて線をつなぐ感覚で読むと理解しやすいです。

## 5. Executor は 1 つの責務に絞る
責務とは、そのステップが何を担当するかという範囲のことです。

たとえば次の 2 つを比べます。

悪い例:
- 1 つの Executor で要約、分類、承認判定、通知まで全部やる

よい例:
- `summarize`
- `classify`
- `check_approval_needed`
- `notify`

分ける利点は次のとおりです。

- 失敗箇所を見つけやすい
- 分岐条件を入れやすい
- 再利用しやすい
- 後から人の確認ステップを挿入しやすい

「1 ステップ 1 概念」で分けるのは、教材だけでなく Workflow 設計でも有効です。

### 要点
- Executor は小さく分けるほど管理しやすいです。
- 1 つの Executor に役割を詰め込みすぎないことが大切です。

## 6. 条件分岐を入れる
条件分岐とは、入力や途中結果によって次の進み先を変えることです。

たとえば問い合わせを処理するとき、分類結果によって担当を変えたいことがあります。

```text
問い合わせを分類
  ├─ 請求なら billing_handler へ
  ├─ 技術なら tech_handler へ
  └─ その他なら triage_handler へ
```

このとき使うのが `Conditional` や `Switch-case` 型の Edge です。  
Microsoft Learn の Edges 解説でも、単純な直列だけでなく、条件つきルーティングが主要パターンとして紹介されています。

考え方のコツは、「分岐条件は Edge に近い場所で読む」と意識することです。  
Executor が何をしたかだけでなく、「その結果でどこへ送るか」まで見て初めて流れがわかります。

### 要点
- 分岐は Edge が担うことが多いです。
- Executor の結果と Edge の条件をセットで読みます。

## 7. 並列実行と集約を理解する
並列実行とは、1 つの入力から複数の処理を同時に進めることです。

たとえば文章を多面的に評価したいとき、次のような構成が考えられます。

```text
原稿
  ↓
fan-out
  ├─ 事実確認
  ├─ 文体チェック
  └─ リスク確認
       ↓
      fan-in
       ↓
    総合レビュー
```

このパターンでは、`Fan-out` で複数 Executor へ広げ、`Fan-in` で再び 1 つにまとめます。  
GitHub の公式 README でも、Workflow は graph-based orchestration を支える機能として紹介されています。  
そのため、単なる順番処理だけでなく、このようなグラフ構造で考えるのが重要です。

並列実行が向く場面:
- 独立した観点で同じ入力を評価したい
- 待ち時間を減らしたい
- 最後に集約担当を置いて総合判断したい

### 要点
- Fan-out は広げる接続です。
- Fan-in は集める接続です。

## 8. 状態を持つとはどういうことか
状態とは、Workflow の途中で覚えておきたい情報です。

たとえば次の情報は状態になりえます。

- 途中の分類結果
- 承認待ちフラグ
- 人から返ってきた回答
- 各 Executor の中間出力

状態が必要になるのは、「後のステップで前の結果を使いたい」ときです。  
単発の関数呼び出しなら引数だけで足りても、途中停止や再開を含む Workflow では状態管理が重要になります。

概念例:

```python
workflow_state = {
    "summary": "...",
    "risk_level": "high",
    "approval_required": True,
}
```

学習の初期段階では、「状態はバケツではなく、次の判断に必要なメモ」と考えると整理しやすいです。  
何でも保存するのではなく、後続ステップに必要な情報だけを残します。

### 要点
- 状態は後続ステップのための情報です。
- 保存しすぎるより、必要最小限を意識します。

## 9. Checkpoint と Resume の考え方
Checkpoint は、Workflow の途中状態を保存して、あとでそこから再開できるようにする仕組みです。

たとえば承認待ちの場面を考えます。

```text
申請内容をチェック
  ↓
人の承認が必要
  ↓
ここで一時停止して保存
  ↓
承認が来たら再開
```

Workflow が長くなったり、人の入力を待ったりする場合、最初からやり直すのは非効率です。  
そのため、途中までの進行状況を保存しておき、`resume` 的な流れで続きから動ける設計が重要になります。

GitHub の公式 README でも checkpointing は Workflows の代表機能として挙げられています。  
初学者のうちは、「長い処理を安全に区切る仕組み」と捉えると十分です。

### 要点
- Checkpoint は途中保存です。
- Resume は保存地点からの再開です。

## 10. Human-in-the-loop を入れる
Human-in-the-loop は、Workflow の途中で人に確認や入力を求める仕組みです。

典型例は次のような流れです。

1. Workflow が内容を確認する
2. 不足情報があると判断する
3. 人に質問する
4. 回答が返ってきたら続きを進める

概念例:

```python
@handler
async def validate_request(ctx, data):
    if not data.get("budget"):
        await ctx.request_info(
            message="予算を入力してください",
            response_type=BudgetResponse,
        )
        return

    return {"validated": True}
```

このコードは、「必要情報が足りないときは先へ進まず、人に問い合わせる」ことを表しています。  
実運用では、承認、補足入力、例外確認などでよく使います。

大事なのは、Human-in-the-loop を「失敗時の後始末」と考えないことです。  
むしろ Workflow に自然に組み込む正式なステップとして扱うと、設計が安定します。

### 要点
- Human-in-the-loop は途中で人に渡す仕組みです。
- 承認や不足情報の回収に向いています。

## 11. Agent を Workflow に入れる
Workflow の各ステップで、Agent を実行したい場面があります。

たとえば:
- 1 つ目の Executor で文章を要約する Agent を呼ぶ
- 2 つ目の Executor でリスク評価 Agent を呼ぶ
- 3 つ目の Executor で結果をまとめる deterministic な関数を使う

このときの見方は次のとおりです。

- Workflow は全体の流れを管理する
- Agent は各ステップで知的な判断を担当する

つまり、Workflow は「司会」、Agent は「担当者」です。  
この分け方をすると、モデルを使う部分と制御部分を切り離して考えやすくなります。

### 要点
- Workflow は流れを管理します。
- Agent はステップ内の判断を担当します。

## 12. Workflow を Agent 的に使う
Microsoft Learn では、Workflow を Agent のように扱う考え方も案内されています。

これは、複数ステップからなる処理全体を 1 つの高機能ユニットとして外から呼ぶイメージです。

たとえば、外側からは次のように見えます。

- 入力: 提案書をレビューしてほしい
- 内部: 要約、観点別評価、承認確認、最終返答
- 出力: 整理されたレビュー結果

外から見ると 1 つの「賢い担当」に見えますが、中では Workflow が動いています。  
この発想は、大きなアプリの中で再利用しやすい単位を作るときに便利です。

### 要点
- 外からは 1 つの機能に見せられます。
- 内部では複数ステップを安定して実行できます。

## 13. どんなときに Workflow を使うべきか
次のような場合は Workflow を使う価値が高いです。

- 処理が 2 ステップ以上ある
- 条件で進み先が変わる
- 複数観点を並列に評価したい
- 途中で人の確認を入れたい
- 長い処理を再開可能にしたい

逆に、単発の質問応答や、ごく短いツール呼び出しだけなら Agent 単体で十分なこともあります。

迷ったときの基準:
- 「流れを図で描きたくなる」なら Workflow 向き
- 「1 回の応答で終わる」なら Agent 単体でもよい

### 要点
- 複数ステップと制御が必要なら Workflow を検討します。
- 単発処理だけなら無理に Workflow 化しなくてもよいです。

## 14. 初学者向けの設計手順
Workflow を作るときは、次の順で考えると進めやすいです。

1. ゴールを書く
2. 中間ステップを 3 から 5 個に分ける
3. 各ステップに 1 つずつ名前をつける
4. どこで分岐するかを決める
5. どこで人の確認が必要かを決める
6. 途中で残す状態を決める
7. 最後に最小構成で動かす

最初から複雑な並列や再開機能を全部入れなくて大丈夫です。  
まずは直列で完成させ、そのあとに分岐や HITL を追加すると失敗しにくくなります。

### 要点
- いきなり全部入りにしないことが大切です。
- 最小構成から育てると理解しやすいです。

## 15. まとめ
- Workflow は、複数ステップの流れを構造化するための仕組みです。
- 基本部品は `Executor`、`Edge`、`WorkflowBuilder` です。
- `Direct`、`Conditional`、`Fan-out`、`Fan-in` を押さえると読める範囲が広がります。
- 状態管理、checkpoint、resume、Human-in-the-loop が実運用では重要になります。
- Agent はステップの中で使い、Workflow は全体の制御を担います。
